# 運作原理

LoreVeil 採「**封包端改寫**」架構：偏好是玩家層級的，隱藏純視覺——真實物品永不改動。本頁說明其內部設計，供維護與除錯參考。

## 偏好儲存

- 持久值存在每位玩家的 PDC（key：`show_detail_lore`，true = 顯示，預設 true）。
- 封包熱路徑可能在 **Netty 執行緒**觸發，不能在那裡碰 Player 或讀 PDC，因此另外維護一份執行緒安全的記憶體快取（`ConcurrentHashMap` 的 UUID 集合），熱路徑只讀它。
- PDC 只在主執行緒讀寫：玩家上線時載入、指令切換時寫入；離線時清除快取避免記憶體洩漏。

## 送出方向（server → client）

攔截 `SET_SLOT` 與 `WINDOW_ITEMS` 封包。若接收者處於隱藏模式，把**有 lore 的物品**換成「顯示副本」：

- lore 只剩一行灰色「物品敘述已隱藏」。
- 副本 PDC 帶兩個欄位：`loreveil:display_only`（標記為顯示副本）與 `loreveil:original_lore`（原始 lore 的完整備份）。
- 備份走 **Adventure 元件 JSON 序列化**（多行以 NUL 分隔），不是舊版 `§` 字串——RGB 顏色、自訂字型、hover 事件經隱藏/還原後完整保留。
- 只操作 clone，真實物品與其 PDC 永遠乾淨。

## 容器格子 vs 背包格子

玩家開著容器（`containerId > 0`）時，同一封包混有「容器格子」和「玩家背包格子」：

- **容器格子**（index < 容器格數）不碰——箱子、自訂 GUI 本來就該顯示完整 lore。
- **玩家背包格子**照樣套用隱藏。
- 容器格數由事件監聽器（`InventoryOpenEvent` / `InventoryCloseEvent`）在主執行緒維護的快取提供；查不到時保守放行整包，寧可多顯示不誤藏 GUI。
- `WINDOW_ITEMS` 的游標物品（carried item）屬於玩家，一律套用隱藏。

## 接收方向（client → server，創造模式）

創造客戶端會把手上物品的副本原封不動回寫伺服器（`SET_CREATIVE_SLOT`）。若那是顯示副本：

- 用內嵌備份**就地還原**成真實物品後放行——創造模式的搬動/複製/丟棄照常運作，寫進伺服器的永遠是真實物品，**污染在結構上不可能發生**。
- 若副本缺備份或備份損毀（不應發生），擋下該封包並重送真實背包。

## 為什麼不直接改物品？

| 方案 | 問題 |
|------|------|
| 改真實物品的 lore | 需備份/還原、與 MMOItems 等插件互踩、丟棄/交易/序列化都是邊界問題 |
| **封包端改寫（採用）** | 真實物品零改動，上述問題全數消失；代價只是每個物品封包多一次 clone |
